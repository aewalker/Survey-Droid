PEOPLES Project Final Report

Initial Planning

In retrospect, our original planning for this project (especially the phone side) seems a little insufficient, but this is largely due to our inexperience with the Android system. It became apparent throughout the course of the semester that some of the features we had originally planned would not be possible given the time constraints of the project, or in some cases, at all. In addition, one of the components (nominally, the surveys themselves) was far more complicated than we had previously assumed, resulting in a ton of collaborative and individual effort towards the end of the semester; at one point Austin moved over to the phone team to deal with this. There was almost no we we could have foreseen this problem, but a little more research at the beginning would have saved plenty of frustration throughout the semester.

Our group was actually lucky in one very important way: we had a real user to tell us what was needed. Naomi has been extremely helpful throughout this project, and we are very thankful for her participation. She was able to tell us what she needed, which essentially pinned down a lot of the functionality of the system right from the start. We didn't have to guess at what a real sociologist would need; we could just ask. She was also able to give us testing phones to use.

One big problem was the order that we choose to do things in; this was an especially big problem on the phone side. We should have done things in almost the opposite order that we did them: we should have defined how things that affected the most other things (for example, communication and background services) should work, and then moved down to the rest of the things. Since we didn't do this, we ended up rushing a lot of Android code that ended up having to be re-written latter on once the code that was going to run above it had been written.

On the web end, Austin and Sema did a great job by using the CakePHP framework, which allowed the web team to keep their project organized and never once had to re-architect the site. Austin already knew CakePHP, which saved the web side from a lot of the troubles that were faced by the phone side.

Milestones

Our progress on our milestones was mixed:

The web team did pretty well keeping up with their milestones. Our milestones worked well on the web side, we didn't have any problem with time. We basically finished the web site as we planned from the very start

The phone team lagged behind. A substantial amount of work on the phone was done the week of the Alpha version and to some extent the week of the Beta version. The problem is not so much that we left those coding tasks to the last minute, but rather we did not anticipate them being needed. In particular, we should have allotted appropriate amount of time to make sure that each and person on the phone team has time to read through Android documentations and walk through the tutorials on the developer site. This learning process really should have been a milestone in and of itself and should have been accounted for in the original project planning.

Taking this into account (the steep Android learning curve), we feel that met met our milestones in an ok but not good way. Our initial milestones were mostly written to break up our final goal into weekly chunks, which doesn't line up perfectly with the time it takes to learn a certain framework.

If we had taken into account the time it would take to learn Android, our phone milestones would have looked better as follows:

  1. learn basics of Android framework: i.e. activities, services, gui stuff
  2. Learn how to obtained the desired data using given libraries
  3. learn how to persist information
  4. Implement GUI, location, call log collection as proper activities/services that rely on separate module for database writing/reading. Write method signatures for any needed database methods.
  5. Implement database handler, and any of the methods that were needed by the activities/services
  6. Integration: work on writing a centralized service that takes care of coordination amongst different modules
  7. Testing

Surprises and Choices

The biggest surprise was the need for more complicated surveys. We were working in the 2nd or 3rd week, and we were looking at the survey Naomi had proposed, and we realized that there was this crazy question that had 3 different branches but only two choices. At that point, we had a full team meeting (including Naomi) and worked all day to figure out how we were going to deal with that. In the end, this surprise turned out ok; the way we dealt with surveys and the system we built up became one of the best parts of the whole project.

Another big surprise was the lack of good 64 bit drivers for the test phones we were given. Even now, the phones do not work very well with most of the group members' computers. This forced us to rely on emulators quite a bit, which made real testing more difficult.

We were also a bit surprised by how difficult it was to coordinate such a large team. Six people is really a pretty huge team. We ended up spending a lot of time and effor just trying to coordinate who did what.

As for decisions: we made a couple of big decisions right at the start. First, we decided to keep the web and phone teams separate to try and make coordination easier. This worked well for the most part, except towards the end of the semester when Austin switched over to help the phone team.

On the phone side, we chose to use PHP and CakePHP over Python and Django. This turned out to be a really great choice because Austin already knew CakePHP. This helped us to avoid a lot of the troubles that the phone side had with learning a new framework.

We also made the choice to use git and GitHug pretty early. This worked out well for the most part once everyone was set up using the system. Some people (especially Vlad) had trouble with git, so it might have been a good idea to have an early meeting about using git and get everyone on the same page.

Conclusion

In the end, everyone enjoyed this project. We didn't get as much done as we had hoped to, but we still learned a lot. It is likely that Naomi will hire a couple of us to work on this project over the summer, so we will have a chance to actually implement the things that we didn't get to this time.

On that note, here are some of the things that we didn't get to do that we wish we had:

  • Phone
    • Track the phone status: when does it get turnned on/off, when is the app disabled?
    • Create a feature that lets the location tracking be disabled automatically when the subject leaves a certain area (Naomi wants this to ensure that she doesn't have data proving parolees violated their parole conditions).
    • Refactor all the code: right now, a lot of the code doesn't fit into packages the way that it naturally should based on its function.
  • Web
    • Create a "drag and drop" interface for creating surveys. This way, the user can build a flowchart/tree, which is the natural way to think about the surveys.
    • Create a "study" level of control: allow multiple studies to be run of one server and be able to set up when they start/stop.
    • Give web users the ability to control more things on the phones via configuration settings.
    • Provide more survey options, specifically, let subjects select multiple answers for a multiple choice question.

If we were to do this again, there are two big things that we could change. First, we would put way more effort into the first stages of the project. We really needed to do more planning. It would have been good to get the group together before Spring Break, but we were one of the later groups to form, so that wasn't possible. Second, we would have liked to have had a smaller team. It's good to have a lot of people to do work, but it's not always worth the effort to coordinate such a large group. We could have been more efficient with fewer people.

Overall, I would say the project had its bumps but we were really able to put everything together toward the end and finish the project and achieve most of our initial targets.
